# PXE operator runbook

How the Homelab PXE stack works, and how to add / remove / refresh
assets in it.

## Architecture, one paragraph

A K8s node PXE-boots over its mgmt-VLAN NIC. MikroTik DHCP hands it
`next-server=10.10.5.10` + `boot-file-name=ipxe.efi`. Node TFTPs
`ipxe.efi` from the `pxe` Custom App on the NAS (a Docker container
running Alpine + dnsmasq + nginx). The iPXE binary's baked-in
script DHCPs again and chains `tftp://10.10.5.10/menu.ipxe`. That
menu offers Talos / BIOS / Utilities / Distributions / Live CDs.
Picking a sub-menu chains a dynamically-generated file listing
every `*.iso` under the corresponding directory on the NAS, plus a
few hand-coded kernel+initrd items (Debian netinst, Alpine
netboot, Memtest). All asset bytes are served as plain static
files from `/mnt/tank/system/pxe/http/` — no proxying, no
redirects, no upstream fetching at boot time.

## Directory layout on the NAS

```
/mnt/tank/system/pxe/tftp/          (bind-mounted /srv/tftp in container)
  ipxe.efi                          custom-built iPXE binary (baked into image)
  menu.ipxe                         top-level menu (hand-written, committed)
  boot.cfg                          runtime vars (hand-written, committed)
  autoexec.ipxe, bios.ipxe,         hand-written sub-menus (committed)
    talos.ipxe, memtest.bin
  utils.ipxe, linux.ipxe,           AUTO-GENERATED by pxe-genmenu.sh
    live.ipxe                       (DO NOT edit — rewritten on every run)

/mnt/tank/system/pxe/http/          (bind-mounted /srv/http in container)
  talos/                            Talos kernel+initramfs (talos-updater cron)
    vmlinuz, initramfs              latest-version symlinks
    vX.Y.Z/                         per-version directories
  talos-menu.ipxe                   rich talos menu (talos-updater cron)
  bios-config/bios-apply.img        from sibling bios-config repo
  extras/                           pre-downloaded PXE assets (pxe-download cron)
    uefishell/Shell.efi
    zbm/zfsbootmenu-recovery.EFI
    debian-13/{linux, initrd.gz}
    debian-12/{linux, initrd.gz}
    alpine/{vmlinuz-lts, initramfs-lts, modloop-lts}
    utils/*.iso                     rescue tools (GParted, Clonezilla, ...)
    distros/*.iso                   server installers (Ubuntu Server, ...)
    live/*.iso                      live CDs (Ubuntu Desktop, Tails, ...)

/mnt/tank/system/apps-config/pxe/
  build/                            container build context (Dockerfile, etc.)
  pxe-download.sh                   downloader (weekly cron + on-demand)
  pxe-genmenu.sh                    menu regenerator (invoked by pxe-download.sh)
  pxe-download.log                  rolling log of download runs
```

## Cronjobs on the NAS

| Description | Schedule | Action |
|---|---|---|
| `talos-updater` | daily 03:00 | Fetches latest Talos release for our schematic; writes into `/pxe/http/talos/`; regenerates `talos-menu.ipxe`. |
| `pxe-download` | Sunday 02:30 | Runs `pxe-download.sh` which curl-fetches every curated asset into `/pxe/http/extras/`, then invokes `pxe-genmenu.sh` to rewrite the dynamic menu files. |

Trigger either on-demand from the laptop:
```sh
midclt -u wss://nas.w1.lv/api/current call cronjob.query  # find IDs
midclt ... call cronjob.run <id>
```

Or via the `truenas-infra` Python client if you prefer scripting.

## Add a new ISO (most common operation)

Two paths. Both take ~1 minute.

### Option A — edit the downloader (preferred for version-tracked assets)

```sh
# In the truenas-infra repo on your laptop:
cd apps/pxe
$EDITOR pxe-download.sh
# Add a `fetch` line under the appropriate section (utils/distros/live).
# Pattern:
#   fetch "<URL>" "$EXTRAS/<category>/<name>.iso" <min-size-bytes>
# e.g.:
#   fetch "https://releases.example.com/foo-1.0.iso" \
#         "$EXTRAS/utils/foo.iso" 100000000

git add pxe-download.sh
git commit -m "pxe: add <tool>"
cd ../..
./manage.sh phase apps --apply    # uploads updated script to NAS

# Trigger the download:
./manage.sh pxe-download          # (if you wire this into manage.sh)
# -- OR --
# Find the cronjob ID:
python -c "
from truenas_infra.config import RuntimeConfig
from truenas_infra.client import connected
s = RuntimeConfig.from_env()
with connected(s.truenas_host, s.truenas_api_key, verify_ssl=s.truenas_verify_ssl) as c:
    cj = c.call('cronjob.query', [['description', '=', 'pxe-download']])
    print('cronjob ID:', cj[0]['id'])
    print('triggered job:', c.call('cronjob.run', cj[0]['id']))
"
```

The new ISO shows up in the menu on the next PXE boot.

### Option B — drop a file directly

You already have an ISO you want to serve? Skip the script:

```sh
scp myrescue.iso root@10.10.5.10:/mnt/tank/system/pxe/http/extras/utils/
ssh root@10.10.5.10 '
    chown 1000:1000 /mnt/tank/system/pxe/http/extras/utils/myrescue.iso
    /mnt/tank/system/apps-config/pxe/pxe-genmenu.sh
'
```

Works for one-offs. For anything you'll want to keep current as
upstream bumps version, use Option A so the URL is version-tracked
in git and the weekly cron re-fetches.

## Remove an ISO

```sh
ssh root@10.10.5.10 'rm /mnt/tank/system/pxe/http/extras/utils/clonezilla-live-3.2.2.iso'
ssh root@10.10.5.10 '/mnt/tank/system/apps-config/pxe/pxe-genmenu.sh'
# Also delete the `fetch` line in pxe-download.sh so the weekly cron
# doesn't re-download it.
```

## Refresh everything

Force a full re-run of the downloader (doesn't re-fetch files that
already exist at the expected size; delete a file first to force
re-fetch):

```sh
# Via API:
midclt ... call cronjob.run <pxe-download-id>

# Or log in to NAS:
ssh root@10.10.5.10 /mnt/tank/system/apps-config/pxe/pxe-download.sh
```

Log goes to `/mnt/tank/system/apps-config/pxe/pxe-download.log`.

## Update an upstream version

Example: Ubuntu ships 24.04.5, we need to update from 24.04.4.

```sh
# Laptop:
cd apps/pxe
$EDITOR pxe-download.sh
# Bump the URL AND the local filename so the size check doesn't
# match the old file:
#   - "https://releases.ubuntu.com/24.04/ubuntu-24.04.4-live-server-amd64.iso"
#   + "https://releases.ubuntu.com/24.04/ubuntu-24.04.5-live-server-amd64.iso"
#
# If you want to keep the same local filename, delete the old file
# on the NAS first so the size-check forces re-fetch:
#   ssh nas rm /mnt/tank/system/pxe/http/extras/distros/ubuntu-server-24.04.iso
git commit -am "pxe: bump Ubuntu Server to 24.04.5"
./manage.sh phase apps --apply
# trigger cron run (see above)
```

## Rebuild the iPXE binary

The iPXE build happens inside the container's Docker build. Forcing
a rebuild:

```sh
# Bump IPXE_REF in apps/pxe/build/Dockerfile (ARG IPXE_REF=master
# → a newer commit SHA). Or edit any other stage to bust the build
# cache. Then:
./manage.sh phase apps --apply    # uploads updated Dockerfile
# On NAS, delete the app's existing image so the next app.start rebuilds:
midclt ... call app.delete pxe '{"remove_images": true}'
./manage.sh phase apps --apply    # recreates, build happens on startup
```

First container start after rebuild takes ~3-5 min (git clone +
compile iPXE); subsequent restarts use Docker's layer cache.

## Troubleshooting

### Node boots, gets an IP, "Homelab iPXE — booting..." then hangs

Likely TFTP can't serve `menu.ipxe`. Check the container:

```sh
midclt ... call app.query | jq '.[] | select(.name=="pxe") | .state'
# Should be RUNNING.
curl -sI tftp://10.10.5.10/menu.ipxe  # should return content
```

Restart if needed:

```sh
midclt ... call app.stop pxe
midclt ... call app.start pxe
```

### Picking a menu item hangs with no error

The asset URL returns a byte stream but slowly. iPXE's `sanboot`
has no progress UI for HTTP, so the screen freezes while data
transfers. Check from the laptop:

```sh
curl -I http://10.10.5.10:8080/extras/<category>/<file>
# Should return 200 with Content-Length matching the expected size.
# If 404: run pxe-download.sh (or check the log for the specific item).
```

### Adding a new distro: how do I find the right URL?

- **GitHub releases** — stable download URL:
  `github.com/<owner>/<repo>/releases/download/<tag>/<filename>`
  (no `/latest/` path since that 302-redirects; iPXE handles via
  sanboot fine once we `curl` it into place on the NAS).
- **SourceForge** — canonical form with `/project/` prefix:
  `downloads.sourceforge.net/project/<proj>/<path>/<filename>`
- **Ubuntu** — `releases.ubuntu.com/<version>/<filename>`
- **Debian** — `deb.debian.org/debian/dists/<codename>/...` for
  netinst kernel+initrd; `cdimage.debian.org/debian-cd/current-live/...`
  for live ISOs
- **Alpine** — `dl-cdn.alpinelinux.org/alpine/v<ver>/releases/<arch>/<file>`

When in doubt, check the sourceforge best-release API:

```sh
curl -s https://sourceforge.net/projects/<proj>/best_release.json | jq
```

### Cache invalidation after bumping a version

`pxe-download.sh`'s idempotency check is: does the file exist at
`>= MIN_SIZE` bytes? It doesn't validate checksum or version. To
force re-fetch of a specific item:

```sh
ssh root@10.10.5.10 rm /mnt/tank/system/pxe/http/extras/<category>/<file>
# Next pxe-download run (or manual trigger) re-fetches.
```

## Related files

- `apps/pxe/menu.ipxe` — hand-written top-level menu
- `apps/pxe/boot.cfg` — runtime vars (site_name, cache_url)
- `apps/pxe/menus/talos.ipxe`, `bios.ipxe`, `autoexec.ipxe` — hand-written
  sub-menus for the Homelab-specific items
- `apps/pxe/pxe-download.sh` — the curated list of upstream URLs
- `apps/pxe/pxe-genmenu.sh` — the dynamic menu generator
- `apps/pxe/build/` — Docker image build context (Dockerfile,
  embed.ipxe, iPXE build-config headers, entrypoint, nginx config)
- `src/truenas_infra/modules/apps.py` — `_ensure_pxe_*_via_ctx`
  helpers (what `./manage.sh phase apps --apply` does)
